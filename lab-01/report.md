## 1. Настройка среды разработки (Docker Compose)

Для работы используется технология контейнеризации Docker. Ниже представлен файл `docker-compose.yml` с подробным разбором каждой директивы.

```yaml
services:
  db:
    image: mysql:8.0
    container_name: mysql-study
    restart: always
    # Добавляем команду запуска с параметрами кодировки:
    command:
      [
        "mysqld",
        "--character-set-server=utf8mb4",
        "--collation-server=utf8mb4_unicode_ci",
      ]
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: lab
    ports:
      - "3306:3306"
    volumes:
      # Связываем именованный том с внутренней папкой MySQL
      - mysql-study-data:/var/lib/mysql
      - ./init-db:/docker-entrypoint-initdb.d

  phpmyadmin:
    image: phpmyadmin:latest
    container_name: phpmyadmin-study
    restart: always
    ports:
      - "8081:80"
    environment:
      PMA_HOST: db
      PMA_USER: root
      PMA_PASSWORD: secret
    depends_on:
      - db

  prisma:
    build: ./prisma
    container_name: prisma-studio
    restart: always
    env_file:
      - ./prisma/.env
    ports:
      - "5555:5555"
    volumes:
      - ./prisma:/app
      - /app/node_modules
    depends_on:
      - db

volumes:
  mysql-study-data:
```

**Разбор конфигурации:**

`services:` — начало описания списка контейнеров, которые будут запущены вместе.

`db:` — сервис базы данных.

`image: mysql:8.0` — используется образ MySQL версии 8.0.

`container_name: mysql-study` — фиксированное имя контейнера для удобства обращения.

`restart: always` — контейнер будет автоматически перезапускаться при сбоях или перезагрузке системы.

`command:` — переопределяет стандартную команду запуска MySQL, принудительно устанавливая кодировку utf8mb4 (поддержка кириллицы и эмодзи) на уровне сервера.

`environment:` — переменные окружения. `MYSQL_ROOT_PASSWORD` задает пароль администратора, а `MYSQL_DATABASE: lab` автоматически создает базу данных с именем `lab` при первом запуске.

`ports: "3306:3306"` — проброс порта: стандартный порт MySQL (3306) внутри контейнера становится доступен на компьютере.

`volumes:`

- `mysql-study-data:/var/lib/mysql` — сохранение данных: даже если удалить контейнер, таблицы останутся в «именованном томе».
- `./init-db:/docker-entrypoint-initdb.d` — важнейшая часть: любой `.sql` файл из локальной папки `init-db` будет выполнен один раз при создании базы.

`phpmyadmin:` — веб-интерфейс для управления базой. `ports: "8081:80"` — доступ через браузер по адресу `localhost:8081`.

`depends_on: - db` — говорит Docker, что веб-морду нельзя запускать раньше, чем будет готов сервер базы данных.

`prisma:` — сервис для ORM (Prisma Studio). `/app/node_modules` — «пустой» путь внутри томов. Это гарантирует, что Docker будет использовать зависимости, установленные внутри образа, а не пытаться найти их в локальной папке проекта.

## 3. Реализация

При выполнении команды `docker compose up -d` запускается сложный процесс оркестрации, который превращает статичные инструкции из конфигурационного файла в живую экосистему изолированных сервисов. Сначала Docker считывает файл настроек и выстраивает граф зависимостей, чтобы определить правильный порядок запуска — это гарантирует, что база данных начнет работу раньше, чем к ней попытаются подключиться Prisma или phpMyAdmin. Параллельно с этим создается виртуальная сеть, которая позволяет контейнерам общаться друг с другом по именам сервисов, полностью скрывая их внутренний трафик от внешней среды, и резервируется место в хранилище под именованные тома для постоянного хранения данных.

Затем Docker переходит к подготовке программной начинки, скачивая готовые образы MySQL и phpMyAdmin из глобального реестра и собирая собственный образ для Prisma на основе локального Dockerfile. Когда образы готовы, наступает фаза активации контейнеров, где происходит самое важное для этой лабораторной работы: если база данных запускается впервые и ее хранилище пусто, MySQL монтирует содержимое локальной папки со скриптами во внутреннюю системную директорию и автоматически исполняет файл `init.sql`. Это действие на лету создает структуру таблиц и наполняет их данными, приводя базу в состояние, которое требуется по условию нормализации.

На финальном этапе Docker связывает внутренние порты контейнеров с портами физического компьютера: через порт `3306` для прямой работы с базой, через `8081` для веб-интерфейса и через `5555` для визуального редактора Prisma.

Скрипт предназначен для автоматического создания структуры таблиц и их первоначального заполнения. Использование `init.sql` позволяет развернуть готовую базу на любом компьютере одной командой.

```sql
SET NAMES utf8mb4;
USE lab;

-- Создаем таблицу отделов (устраняем транзитивность - 3НФ)
CREATE TABLE IF NOT EXISTS Departments (
    dept_id INT PRIMARY KEY,
    dept_name VARCHAR(255) NOT NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Создаем таблицу сотрудников
CREATE TABLE IF NOT EXISTS Employees (
    emp_id INT PRIMARY KEY,
    full_name VARCHAR(255) NOT NULL,
    position VARCHAR(255) NOT NULL,
    dept_id INT,
    FOREIGN KEY (dept_id) REFERENCES Departments(dept_id) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Создаем таблицу квалификаций (обеспечиваем атомарность - 1НФ)
CREATE TABLE IF NOT EXISTS EmployeeQualifications (
    id INT AUTO_INCREMENT PRIMARY KEY,
    emp_id INT NOT NULL,
    qual_name VARCHAR(100) NOT NULL,
    FOREIGN KEY (emp_id) REFERENCES Employees(emp_id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- Заполнение данными из задания
INSERT INTO Departments (dept_id, dept_name) VALUES
```

### Разбор Dockerfile: Пошаговая логика сборки

```dockerfile
FROM node:22-slim

# Устанавливаем системные зависимости для Prisma
RUN apt-get update && apt-get install -y openssl && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# Сначала инициализируем пустой проект, чтобы создать package.json
RUN npm init -y

# Теперь устанавливаем зависимости
RUN npm install prisma@latest @prisma/client@latest dotenv

# Открываем порт для Studio
EXPOSE 5555

# Запускаем Prisma Studio
CMD npx prisma studio --port 5555 --browser none
```

Инструкции внутри Dockerfile выполняются строго последовательно, создавая слои образа, которые Docker кэширует для ускорения последующих запусков.

Процесс начинается с выбора фундамента в директиве `FROM`, где указывается базовый образ операционной системы с уже предустановленной средой выполнения — в данном случае это легковесная версия Node.js 22. Далее следует этап подготовки системного окружения с помощью команд `RUN`, где обновляются индексы пакетов и устанавливаются критические библиотеки, такие как OpenSSL, необходимые для работы криптографических функций Prisma.

Установка рабочей директории через `WORKDIR` создает изолированное пространство внутри контейнера, аналогичное корневой папке проекта. Команды инициализации проекта и установки зависимостей через `npm` формируют программный слой приложения, подтягивая актуальные версии Prisma и клиента.

Завершается сборка декларацией порта через `EXPOSE`, которая носит информационный характер для Docker, и финальной командой `CMD`. В отличие от всех предыдущих шагов, `CMD` не выполняется в момент сборки образа, а служит инструкцией по умолчанию, которая сработает только при непосредственном старте контейнера.

### Процесс полной переустановки: «Снести и создать заново»

Когда в проекте происходят радикальные изменения версий (например переход с Node.js 18 на 22 или с MySQL 5.7 на 8.0), обычного перезапуска недостаточно, так как Docker старается максимально использовать старые данные для экономии времени.

Для гарантированно чистой установки требуется выполнить глубокую деструкцию текущего состояния. Первым делом необходимо остановить все процессы и физически удалить созданные контейнеры вместе с их виртуальными сетями и именованными томами данных с помощью команды `docker compose down -v`. Флаг `-v` критичен: без него база данных не обновится, так как старые файлы MySQL останутся на диске и заблокируют инициализацию новой версии.

![Рисунок – Команда docker compose down -v](assets/cmd-down.png)

Следующим этапом является очистка кэша сборщика через команду `docker builder prune -f`, которая позволяет системе освободить ресурсы, удалив неиспользуемые и «осиротевшие» (orphans) данные сборки.

![Рисунок – Команда docker builder prune -f](assets/cmd-prune.png)

Финальный запуск через `docker compose up --build -d` инициирует создание системы с абсолютно чистого листа: Docker заново скачивает образы, выполняет каждую строчку Dockerfile, прогоняет скрипты инициализации базы данных и поднимает сервисы в их актуальном состоянии.

### Механизмы миграции и синхронизации схемы в Prisma ORM

В процессе управления архитектурой базы данных через Prisma ORM ключевую роль играет механизм отслеживания изменений, инициируемый командой `npx prisma migrate dev`.

![Рисунок – Команда npx prisma migrate dev](assets/prisma-migrate-dev.png)

Данная директива предназначена для генерации версионированных SQL-скриптов на основе любых изменений в файле локальной схемы и их строгого последовательного применения к целевой базе. При использовании комбинированного подхода, когда первичная структура таблиц создается внешними скриптами инициализации, возникает конфликт состояний. Для разрешения подобных ситуаций применяется команда `npx prisma migrate reset`, которая радикально очищает текущую базу данных вместе со всей конфликтующей историей миграций.

![Рисунок – Команда npx prisma migrate reset](assets/prisma-migrate-reset.png)

После стабилизации рабочего окружения для выравнивания состояний между кодом и реальной базой данных применяется команда обратного инжиниринга `npx prisma db pull`.

![Рисунок – Команда npx prisma db pull](assets/prisma-db-pull.png)

Процесс интроспекции заключается в том, что система подключается к активной базе данных, сканирует существующие таблицы, связи и типы колонок, после чего автоматически генерирует соответствующие модели внутри локального файла `schema.prisma`.

В противоположность процессу чтения структуры применяется команда форсированной записи `npx prisma db push`, которая транслирует локальные изменения напрямую в ядро базы данных. В отличие от стандартных миграций, этот подход не создает промежуточных файлов истории, агрессивно перезаписывая схему на сервере.

![Рисунок – Команда npx prisma db push](assets/prisma-db-push.png)

Ниже приведена структура схемы, которая трансформируется в базу данных через этот механизм:

```prisma
generator client {
  provider = "prisma-client-js"
  output   = "../generated/prisma"
}

datasource db {
  provider = "mysql"
  url      = env("DATABASE_URL")
}

model Departments {
  dept_id   Int         @id
  dept_name String      @db.VarChar(255)
  Employees Employees[]
}

model EmployeeQualifications {
  id        Int       @id @default(autoincrement())
  emp_id    Int
  qual_name String    @db.VarChar(100)
  Employees Employees @relation(fields: [emp_id], references: [emp_id], onDelete: Cascade, onUpdate: Restrict, map: "EmployeeQualifications_ibfk_1")

  @@index([emp_id], map: "emp_id")
}

model Employees {
  emp_id                 Int                      @id
  full_name              String                   @db.VarChar(255)
  position               String                   @db.VarChar(255)
  dept_id                Int?
  EmployeeQualifications EmployeeQualifications[]
  Departments            Departments?             @relation(fields: [dept_id], references: [dept_id], onUpdate: Restrict, map: "Employees_ibfk_1")

  @@index([dept_id], map: "dept_id")
}
```
